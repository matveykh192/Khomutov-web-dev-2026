# ЛР5 — авторизация и статистические отчёты

Доработка ЛР4: добавлены роли с проверкой прав через декоратор `check_rights`,
журнал посещений с автоматическим заполнением через `before_request`,
отчёты по страницам и пользователям (с экспортом в CSV), Blueprint для отчётов.

## Структура

```
lab5/
├── app.py              # основное приложение, routes пользователей, before_request
├── auth.py             # декоратор check_rights
├── models.py           # SQLAlchemy-модели: User, Role, VisitLog
├── reports.py          # Blueprint 'reports' с журналом и отчётами
├── test_app.py         # тесты (37 шт.)
├── pytest.ini
├── requirements.txt
└── templates/
    ├── base.html
    ├── index.html
    ├── login.html
    ├── change_password.html
    ├── macros.html
    ├── user_form.html
    ├── user_view.html
    └── reports/
        ├── index.html      # журнал посещений с пагинацией
        ├── by_pages.html   # отчёт по страницам
        └── by_users.html   # отчёт по пользователям
```

## Запуск

```bash
python3 -m venv venv
source venv/bin/activate          # Windows: venv\Scripts\activate
pip install -r requirements.txt
python app.py
```

При первом запуске автоматически создаются роли, админ и тестовый пользователь.

## Реализация

* **`check_rights(action)`** в `auth.py` — обёртка над view-функцией.
  Достаёт `id`/`user_id` из kwargs, вызывает `current_user.can(action, target_user)`.
* **`User.can(action, target_user)`** в `models.py` — единое место с правилами.
  Админ может всё; обычный пользователь — только `show`/`edit` для самого себя.
* **`@app.before_request log_visit`** в `app.py` — на каждый GET-запрос (кроме
  `/static/*` и `/favicon.ico`) добавляет запись в `visit_logs` с `user_id`
  текущего пользователя (или `NULL`).
* **Blueprint `reports`** в `reports.py` подключается через
  `app.register_blueprint(...)` и реализует журнал (с пагинацией), два отчёта и
  два маршрута экспорта CSV (с UTF-8 BOM, чтобы Excel корректно открывал
  кириллицу).
* В шаблоне `index.html` кнопки действий отображаются только если
  `current_user.can(...)` возвращает True.
* В форме редактирования (`macros.html`) поле `<select name="role_id">`
  получает атрибут `disabled`, если текущий пользователь не админ. На бэкенде
  смена роли тоже игнорируется, если пользователь не админ (защита от подмены
  через DevTools).
